# Design Docs スキル

プロジェクトの `.design/` ディレクトリ配下のドキュメントを管理するスキル。

## ドキュメント構成と役割

| ファイル | 役割 | 更新タイミング |
|----------|------|----------------|
| `granddesign.md` | プロジェクト全体のビジョン・目的・ゴール | プロジェクト開始時、方向性変更時 |
| `architecture.md` | システムアーキテクチャ・構成図・技術スタック | 設計変更時、新コンポーネント追加時 |
| `guidelines.md` | コーディング規約・命名規則・ベストプラクティス | ルール追加・変更時 |
| `decisions.md` | 技術選定・設計判断の記録（ADR形式） | 重要な意思決定時 |
| `tasks/` | タスク管理・進捗追跡（個別ファイル形式） | 作業開始・完了時 |
| `non-blocking-issues.md` | 後回しにする問題・技術的負債の記録 | 問題発見時、解決時 |

## 各ドキュメントの詳細

### granddesign.md
プロジェクトの「なぜ」を記録する。
- プロジェクトの目的・背景
- 解決したい課題
- 成功の定義
- スコープ（やること・やらないこと）

### architecture.md
システムの「何」と「どのように」を記録する。
- システム構成図
- 主要コンポーネントの説明
- 技術スタック
- データフロー
- 外部連携

### guidelines.md
開発の「ルール」を記録する。
- コーディング規約
- 命名規則
- ディレクトリ構成
- コミットメッセージ規約
- レビュー基準

### decisions.md
意思決定の「経緯」を記録する（ADR形式推奨）。
```markdown
## YYYY-MM-DD: [決定事項のタイトル]

### 背景
[なぜこの決定が必要だったか]

### 選択肢
1. [選択肢A] - [メリット/デメリット]
2. [選択肢B] - [メリット/デメリット]

### 決定
[選んだ選択肢と理由]

### 影響
[この決定による影響]
```

### tasks/
タスクを個別ファイルで管理する。`YYYYMMDD-title.md` 形式で作成。
```markdown
# [タスクタイトル]

## ステータス
- [ ] 未着手 / [x] 進行中 / [x] 完了

## 概要
[タスクの説明]

## 完了条件
- [ ] 条件1
- [ ] 条件2
```
詳細は `.design/tasks/README.md` を参照。

### non-blocking-issues.md
後回しの問題を記録する。
```markdown
## [問題のタイトル]
- **発見日**: YYYY-MM-DD
- **重要度**: 低/中/高
- **説明**: [問題の詳細]
- **対応案**: [可能な対応策]
- **ステータス**: 未対応/対応中/解決済み
```

## 使い方

### ドキュメントを更新する場合
1. 適切なファイルを特定する（上記表を参照）
2. 既存の内容を確認する
3. フォーマットに従って追記・更新する
4. 更新日時を記録する（必要に応じて）

### 新しいドキュメントを作成する場合
`.design/` ディレクトリが存在しない場合は `vc` コマンドで初期化する。
